# Day7（中間）・Day14（最終）— Go 条件と投票ルール

[MVP_CHECKLIST.md](./MVP_CHECKLIST.md) の**必達**が最終的な合格ライン。ここでは**中間でブレを止める**ための Go 条件と、**コミュニティ最終日**の投票ルールを定義する。

## Day7 — 中間発表（Go / No-Go）

**目的**: アイデアと実装方針の解像度を上げ、**最終 1 週間の無駄を減らす**。採点ではない。

**第1期の到達ライン（インタビュー反映）**: 中間時点では **G4 まで揃っていればよい**（**G5** は最終週に向けて推奨。指示テンプレの解像度は **Day7 で上げきれなくても**、**良い指示書が出せるレベル**を目指す）。

### Go（このまま実装してよい）

| # | 条件 |
|---|------|
| G1 | **誰向けの何の問題**か一文で言える |
| G2 | **主導線が 1 本**（ユーザが最初にやること → ゴール）が言語化されている |
| G3 | **ログイン OR DB** のどちらで満たすか決まっている |
| G4 | **画面が 2 枚分以上**ラフでもよいので見える／説明できる |
| G5 | **Cursor／LLM** への指示に **再利用できる型**（テンプレ）が 1 つある — **ハーネス思考**のチェック |

### No-Go（戻してよい）

- 上記が揃わない → **範囲を切る**（機能を減らす／ログインのみにする等）をメンターと合意してから再提出

### 提出物（例）

- 画面ラフ（手書き・Figma・スライドどれでも可）
- データの保存場所メモ（どのテーブル／どのユーザー属性か）
- 指示テンプレ 1 枚（コピペ用）

## Day14 — 最終発表と投票

### デモ形式（第1期・インタビュー反映）

- **発表 5 分** + **フィードバック／質疑 5 分**（Day7 と同じ枠で統一）
- **スライド**あり（受講生が用意。**公開デモ**まで実施）
- **公開 URL**・**リポジトリリンク**（または提出場所）を事前共有

### 投票の目的

「優勝」を楽しむための**仲間内評価**。採用・学業の正式成績ではない。

### 評価観点（投票時に伝える）

| 観点 | 意味 |
|------|------|
| **改善ループ** | 友達などに見せてフィードバックを取り、反映したか（証拠は任意提出：Before/After、コミット、メモ） |
| **実用性** | 誰かの役に立ちそうか／面白いか（主観でよい） |

### 投票の構成（第1期）

| 票の種類 | 第1期の前提 |
|----------|-------------|
| **受講生票** | コミュニティ内の仲間評価 |
| **メンター票** | メンター陣 |
| **企業票** | **協賛・ゲスト企業がいない前提では設けない**（**企業 0 のときは受講生票・メンター票のみ**） |

**受講生 1 票とメンター 1 票の価値**は**同じ**（重みづけしない）。

**每人の投票数**（本人以外に何票まで等）は**開催前に**運営で確定し、当日冒頭で宣言する。

### 運営ルール（第1期・確定）

1. **每人 投票数**を決める（例: 本人以外に 3 票）
2. **自分票・班の固定票**は禁止（わかる範囲で）
3. **同票時**は**併優**とする（**同率トップはともに優勝**）

### 企業向けに添える一文（任意）

投票結果は**学習コミュニティ内の人気**であり、**プロダクトの市場価値の証明ではない**。改善の習慣と発表力の可視化が目的。
