# メンター運用ハンドブック（火・木オフィスアワー）

**時間帯（初回コホート）**: **19:00〜21:30**（[`FIRST_COHORT_SCHEDULE.md`](FIRST_COHORT_SCHEDULE.md)）。期ごとに前後する場合は FAQ で告知。

**前提**: コアメンター 2〜4 名＋当日ボランティア。全員に常時 1 対 1 はしない。**班（8〜10 名×数）＋班長**で相互解決を先に回す。

### Discord の「3 チャンネル」の意図（[`DISCORD_NAMING_AND_CHANNELS.md`](DISCORD_NAMING_AND_CHANNELS.md)）

班チャットと混同しやすいので、**サーバー全体で最低限ある 3 本**の役割を固定する。

| チャンネル | 意図 |
|------------|------|
| **`#質問`（1本）** | **技術・環境・デプロイ**の**チケット正**。検索・公平性・抜け漏れ防止のため**ここに集約**。 |
| **`#自己紹介`** | **所属とやりたいこと**を一言でそろえ、**班や OH で声をかけやすくする**（コミュニティの土台）。 |
| **`#雑談`** | **非技術の軽い連絡**。**質問は #質問へ誘導**し、**#質問**を読みやすく保つ。 |

**班チャット**（班単位のスレ／スレッド）があれば、**班内の雑談・日程調整・相互教え**用。**技術の正**は引き続き **`#質問`**（運用の住み分けは開講前に一文で告知）。

### 勉強会（まくり・Day13）の位置づけ（インタビュー反映）

- **場だけ**提供し、**メンターが参加**。  
- **質問があれば随時**。  
- **一人ひとりを見る人**がいるような **1on1 寄り**（呼び出しで巡回）。  
- 詳細は [`DAY1_DAY7_DAY14_CONTENT.md`](DAY1_DAY7_DAY14_CONTENT.md) の **Day13**。

### 赤・黄のフォローと「達成率」

- **赤・黄**（スコープ）の受講生には **個別フォロー**を**文言中心**で（DM／短いテンプレ）。  
- **進捗の見方（案）**: まず **2 週間の受講生スケジュール**（[`student_ops_sheet_spec.md`](../operations/student_ops_sheet_spec.md)）を整備し、**チェック項目が並んだうえで**「**達成率**」の**計算式**を運営が確定する。

## 基本方針

| 原則 | 内容 |
|------|------|
| 同期 | 質問は必ず**テンプレ**で投稿（環境／デプロイ／仕様／バグ） |
| SLA を約束しない | 「その場で必ず解決」は言わない。**24〜48 時間以内に次の一手**を返す目安で運用 |
| エスカレーション | コアが優先するのは **デプロイ・認証・課金・秘密情報漏えい疑い**。UI の好みは班内 |
| 火木の位置づけ | 「講義」ではなく**クリニック**（詰まりの除去） |
| 録画 | **原則しない**（[`LP_AND_SPONSOR_RULES.md`](LP_AND_SPONSOR_RULES.md)） |

## チケット制（推奨フォーマット）

受講生は **Discord 上の指定チャンネル（1 箇所）**に、以下をコピペして投稿する（**第1期の正**。[`COMMUNITY_AND_COMMS.md`](COMMUNITY_AND_COMMS.md)）。**メンター DM は可だが推奨しない**。

```
【カテゴリ】環境 / デプロイ / 仕様 / バグ / その他
【いまの状態】（一言）
【やりたいこと】（一言）
【試したこと】（箇条書き）
【スクショ or エラー全文】（可能なら）
【標準スタックか】はい / いいえ（いいえなら構成）
```

**優先度（メンターが見る順）**

1. 秘密情報がリポジトリに入っている疑い → **即対応**
2. デプロイできず URL が出ない
3. 認証・DB 接続エラー
4. 仕様・バグ（再現手順があるもの）

## 班サポートの流れ（各セッション冒頭 5 分）

1. 班長が「班チャットで解決した質問」を共有（重複防止）
2. メンターは**チケット一覧**から上から処理
3. 同じ質問が 3 回目 → **FAQ に 1 行追加**（教材オーナーへ連携）

## メンター用ショート回答テンプレ

- **まず標準レーンに寄せられる？** → [STACK_CHOICE.md](./STACK_CHOICE.md) を見せる
- **鍵が repo にない？** → `.env` / `.gitignore` / Vercel の Environment Variables
- **デプロイログの見方** → ホストのビルドログを貼ってもらう

## やらないこと

- 受講生の代わりに**丸ごとプロダクト仕様を決め打ちしない**（MVP 定義は受講生本人・Day1 シート）
- 徹夜前提のサポートを促さない
