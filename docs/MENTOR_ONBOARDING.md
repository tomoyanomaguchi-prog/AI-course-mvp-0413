# メンター育成・オンボーディング（コア1＋業務委託大学生〜10・未経験寄りのとき）

**課題**: 候補はいるが **ツール未使用レベル** が多く、**育成コスト**と**戦力化の定義**を先に置かないと運営が不安定になる。

**方針**: 全員を「即コア」にしない。**ティア（段階）**と**到達チェック**を決め、開講前に **最低限のラインだけ**揃える。足りない分は **募集人数・OH回数・チケット運用**で吸収（[`ORG_ROLLOUT_FLOW.md`](ORG_ROLLOUT_FLOW.md)）。

**本コホートの人数目安（2026・インタビュー反映）**: **主催＋社員1名＋業務委託2名**で**運用上足りる**想定（**計約4名**）。**業務委託2名が揃わない場合**は **社員2名でオペ・OH をカバー**する（インタビュー反映）。それでも不足時は下の「つまみ」へ。

---

## 1. メンターのティア（役割の切り分け）

| ティア | 役割 | 想定人数（例） |
|--------|------|----------------|
| **T0 コア** | 優先度判断・エスカレーション・デプロイ/認証の最終窓口・運営連絡 | **社員1＋信頼できる委託1〜2**（最大3） |
| **T1 OH** | 火木のクリニック・チケット対応・**標準レーン**の質問（[`STACK_CHOICE.md`](STACK_CHOICE.md)） | 委託から **4〜6** を目標 |
| **T2 補助** | 班チャットの雑談対応・資料の案内・**解決しない質問のタグ付け** | 残りはここからスタートでも可 |

**ルール**: T1 は「**答えられなくてよい**。エスカレーションの型が使えること」が必須。T0 がいない時間帯は **スコープを絞る**（黄を切る・FAQ に回す）。

## 2. 育成で「最低限やること」（学習目標）

### 全員共通（T1/T2）

- [`MVP_CHECKLIST.md`](MVP_CHECKLIST.md) を読み、**何が「卒業」か**説明できる  
- [`IDEATION_AND_SCOPE_LADDER.md`](IDEATION_AND_SCOPE_LADDER.md) で **緑／黄／赤** を使い、受講生に**範囲を切る会話**ができる  
- [`mentor_ops_handbook.md`](mentor_ops_handbook.md) の **チケットテンプレ**を使う  
- **秘密情報**（APIキー・`.env` コミット）のNGを口頭で言える  

### T0 追加

- Vercel / Supabase の **環境変数・デプロイ失敗ログ**の読み方（入口レベルでよい）  
- 「**標準レーン外は自己責任**」の線引き  

### ツール体験（推奨順）

1. **GitHub**（リポジトリ作成・clone・push のイメージ）  
2. **標準レーンのチュートリアル1本**（公式 or 社内10〜15分動画）— **自分で1回デプロイ**までやらせると強い  
3. **Cursor**（[`STACK_CHOICE.md`](STACK_CHOICE.md)）— サンプルリポを開き、**AI と1ファイル修正**→保存まで **30分**（学生向け Pro 条件は [`POLICY_SCALE_AND_COSTS.md`](POLICY_SCALE_AND_COSTS.md)）  
4. **Manus** — **v1 では受講生に使わない**ため、メンター研修でも**省略可**（v2 で採用時に [`manus_reference.md`](manus_reference.md) を追加）  

**完璧不要**: メンター全員がフルスタックにならなくてよい。**地図と避難経路**がわかれば T1 は回る。

## 3. 研修の形（育成コストを抑える）

| 形式 | 内容 | 主催の負担目安 |
|------|------|----------------|
| **同期オンボーディング** | 週1×60分×**3〜4回**（開講前） | スライド1本化して再利用 |
| **ハンズオン課題** | 「サンプルリポを fork → 環境変数ダミー → Vercel デプロイ」**同じ手順を全員** | 手順書1枚（[`SETUP_CHECKLIST.md`](SETUP_CHECKLIST.md) 拡張でも可） |
| **メンター用ミニFAQ** | よくある10問（デプロイ・CORS・Supabase RLS 入門など） | 追記型で育てる |
| **シャドウ** | 開講後1週間、T1 は **T0 の横で**1回だけ見学 | スケジュールだけ確保 |

**業務委託の契約上**: 「研修時間」「OHの最低回数」「エスカレーション先」を**書面で揃える**と揉めにくい。

## 4. スケジュール感（開講日を T とする）

目安。**開講4週前**から動き出すと余裕がある。遅くても **T-2週** には T0/T1 のラインを確定したい。

**初回コホート（2026年5月）**: **T-2週末の到達チェック = 4月26日（土）**（[`FIRST_COHORT_SCHEDULE.md`](FIRST_COHORT_SCHEDULE.md)）。**業務委託メンターの内定・契約目線も 4/26 まで**でよい（インタビュー反映）。

| 時期 | やること |
|------|----------|
| **T-4週** | キックオフ（全員）：コースのゴール・ティア説明・資料URL一括送付 |
| **T-3週** | ハンズオン1：GitHub + ローカル or テンプレ起動（T0がデモ） |
| **T-2週** | ハンズオン2：Vercel + Supabase **ダミープロジェクト**でデプロイまで（T1必須） |
| **T-2週末** | **到達チェック**（下記）→ T1 認定 or T2 に留める |
| **T-1週** | Cursor 体験30分 + メンターFAQ読み合わせ + OHの役割プレイ（ロールプレイ15分） |
| **T-1週** | コアだけ **Day0/Day1 の進行リハ**（任意だが効く） |
| **開講後Week1** | 振り返り15分（何が多かったか→FAQに1行） |

**時間の目安（1人あたり）**: 研修 **4〜6時間** + 初週の余韻 **1〜2時間**。コアは **+2〜4時間**。

---

## 5. 到達チェック（T1 になってよいライン）

開講前に **チェックリスト**で判定。全部いなくてもよいが、**半分以下なら T2**。

- [ ] 標準レーンの名前（**Cursor** / Next / Vercel / Supabase）を言える  
- [ ] 「ログイン or DB のどちらで卒業するか」を受講生に聞き返せる  
- [ ] チケットに **カテゴリとスクショ** を依頼できる  
- [ ] APIキーをrepoに貼った人に **即エスカレーション**できる  
- [ ] 黄のテーマを見て「**緑に切る提案**」を1つできる（[`IDEATION_AND_SCOPE_LADDER.md`](IDEATION_AND_SCOPE_LADDER.md)）  

T0 は上に加え、デプロイログを**一通り**見た経験があること。

---

## 6. 育成コストが足りないときのつまみ（優先順）

1. **募集上限を下げる**（40→30 等）  
2. **OHを週1に減らす**（火のみ等）— カリキュラムと [`合同イベント_商品群_マスタ.csv`](../operations/合同イベント_商品群_マスタ.csv) を要修正  
3. **T1 を4人に絞る**、残りは T2（班サポートのみ）  
4. **FAQ・動画を増やして**チケットを減らす
