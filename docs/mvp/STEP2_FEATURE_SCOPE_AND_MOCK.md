# 機能の「個数」と「難易度」、モックに何を付けるか

**流れの位置づけ**（[`STEPS_MVP_FLOW.md`](STEPS_MVP_FLOW.md)）: **個数・難易度・除外**は主に **Step2（ストーリー `STEP2_STORY_SCENARIO.md`）** で決める。**モックに何を付けるか**の線引きは **Step3（`STEP3_UI_MOCK.html`）** を書くときに使う。

**Step2 の意図（運用上の主メッセージ）**

- **機能の名前をべらぼうに増やさない**（主導線に直結しないものは「書かない／後回し」と書く）。
- あわせて、**1つだけでも重い機能**（リアルタイム双方向チャット、本格権限、深い外部連携など）があると、個数を少なくても週が足りなくなる。

つまりマネジメント対象は **二軸** です。

| 軸 | 何が問題になりやすいか | Step2 でやること |
|----|------------------------|------------------|
| **個数** | 画面・導線・「やりたいこと」が増えすぎる | **核になる機能名を 1〜3 個**に言語化し、あとは「最終週」「削る」とラベルする |
| **難易度（1機能あたり）** | 黄・赤に近い要素が混ざる（[`IDEATION_AND_SCOPE_LADDER.md`](../IDEATION_AND_SCOPE_LADDER.md)） | その機能を **緑に落とす案**（削る・簡略化）か **諦める宣言** を書く |

「個数だけ少なければ OK」ではなく、**少ない個数の中身が軽いか**もセットで見る、がここのメインです。

---

## モックで「画面切り替えまで」付けてよいもの（サンプル品質の目安）

**目的**: 見た目と主導線の合意。**本物のデータ・認証・永続化は不要**。

| モックに付けてよいもの | 例 | 備考 |
|------------------------|-----|------|
| **画面の切り替え** | `display` の切替、同一 HTML 内の `#screen-a` / `#screen-b` | 既存サンプル（[`mocks/`](mocks/)）と同レベルで十分 |
| **ダミーの一覧・表・カード** | 行は固定5件でよい | 「見る」だけなら静的でよい |
| **ダミーのフォーム入力欄** | 入力しても送信しない、または `alert` のみ | バリデーション不要 |
| **ログインの「見た目」** | メール欄＋パスワード欄＋ボタン → クリックで次画面へ | **本物のログインではない**。あくまでワイヤ |
| **ダッシュボードの「見え」** | KPI の数字は固定文字、グラフは矩形の塗りでもよい | **動的集計はしない** |
| **保存ボタン（演出）** | `alert('モック')` や、次画面への遷移だけ | **DB に保存しない** |

---

## モックに「付けるとキツくなる」もの（付けるなら意味を限定する）

「つけられない」ではなく、**つけたと勘違いされると最終週がパンクする**ものです。

| 機能の例 | モックでやりがちなこと | 本番で実装するときの論点（難易度） |
|----------|-------------------------|--------------------------------------|
| **ログイン（本物）** | 見た目だけなら OK | Supabase Auth 等は **緑内でも学習コストと時間を食う**。Step2 では「ログインで満たす／DB で満たす」の **方針だけ**（Step1 G3）で止め、画面はダミーでよい |
| **ダッシュボード（本物）** | 静的ダミーなら OK | **集計・権限・複数ロール**が入ると黄寄り。**見るだけ・1ユーザー・単一テーブル**なら緑 |
| **チャット** | 吹き出しの並びを静的表示なら OK | **リアルタイム双方向**は [`IDEATION_AND_SCOPE_LADDER`](../IDEATION_AND_SCOPE_LADDER.md) の **黄**。個数が少なくても重い |
| **保存（本物）** | 演出のみなら OK | **永続化**は必達（[`MVP_CHECKLIST.md`](../MVP_CHECKLIST.md)）で **最終週の仕事**。Step2 の HTML では DB に繋がない |

---

## 問題になるのは「個数」か「難易度」か → **両方**

- **個数**: 機能名が10個あると、主導線が分岐し、説明も実装も破綻しやすい。
- **難易度**: 機能が **1つだけ** でも「常時接続チャット」「決済」「本格 RBAC」なら、2週間の枠と教材レーンから外れやすい。

Step2 のドキュメント（`STEP2_STORY_SCENARIO.md`）と Step3（任意 `STEP3_UI_MOCK.html`）では、

1. **核機能を 1〜3 個に書き切る**  
2. **黄になりうる要素を1つでも列挙し、削るか簡略化するか決める**  

までがゴール、と整理すると運用と合います。
