# Step2 — スコープ・画面ラフ・発表たたき（メンター提出用）

## 入力の要約（Step1 より）

- **G1**: 28卒・サマーインターン応募が増える自分向けに、**散在する過去のES・メモを材料に、新規企業向けのES「叩き」を素早く作れる**ようにする。
- **主導線（G2）**: 応募企業を**追加するか一覧から選ぶ** → **材料を貼る／選ぶ** → **叩きを生成（LLM）** → **企業単位で保存** → 一覧から再開。
- **G3**: **DB**（ユーザー単位で材料・叩き・応募企業を永続化）。デモは**架空・ダミーテキスト**（`MVP_CHECKLIST` 整合）。
- **帯**: **緑**（一覧／詳細で3枚以内、GDoc連携・選考管理は捨てる）。

---

## スコープ案（3〜4段）

| 案 | 概要 | 画面数目安 | データ | 1週間リスク | 推奨 |
|----|------|-------------|--------|-------------|------|
| **A 最小** | 応募企業は**手入力1件固定**または**ローカル状態のみ**で、まず **材料貼付 → 叩き生成 → 1行保存** だけ通す。一覧は後回し。 | **1〜2**（同一画面に結果表示まで） | 単一テーブル or 最小CRUD | **G4（2画面説明）が弱くなりやすい**／「企業がどんどん増える」要件とズレる | □ |
| **B 標準** | **応募企業一覧**（追加）＋**企業詳細ワークスペース**（材料・生成・叩き保存）。教材どおり **Next + Vercel + Supabase**。 | **2**（一覧／詳細） | `companies` 等に企業名・材料テキスト・叩き本文・`user_id` | LLM API・環境変数・エラー表示のバランス。許容範囲内 | ☑ |
| **C ひと伸び** | B に **企業名の簡易検索** または **材料の「よく使う断片」を1ブロック保存**（再利用）。 | **2**（検索は一覧内） | B + カラム追加 or 小テーブル | **検索・断片管理で1週間が膨らむ**。黄ではないが優先を誤ると核が遅れる | □ |
| **D 参考（非推奨・2週間後半〜）** | Googleドキュメント連携、締切・選考ダッシュボード、設問フォーマット完全対応。 | **4+** | 外部API・複数エンティティ | **黄〜赤寄りの重さ**、主導線分岐 | □ |

**推奨**: **B（標準）** — Step1 の核（**企業の随時追加・一覧から再開**）と一致し、**G4 を2画面で満たしやすい**。永続化は **Supabase のテーブル＋Auth** で必達の「DB／ログイン」を教材レーンどおり満たせる。

## 必達との対応（MVP_CHECKLIST）

| 必達 | この推奨案で満たし方 |
|------|----------------------|
| **公開URL** | Vercel に Next をデプロイし、HTTPS の URL を提出。 |
| **核機能1本道** | 一覧で企業を選ぶ／追加 → 詳細で材料 → 叩き生成 → 保存。分岐は最小。 |
| **ログイン or DB** | **Postgres に企業・材料・叩きを保存**（Step1 の G3）。Supabase Auth で本人のみが自分の行にアクセス。 |
| **レスポンシブ最低限** | 一覧・テキストエリアはモバイル幅で横スクロール／折り返し確認。 |
| **README** | 起動、`env` のキー名（値は書かない）、Supabase マイグレーション手順、既知の制限。 |
| **秘密情報** | API キー・サービスロールは **`.env` / ホストシークレット**のみ。デモはダミー文章。他人のPI・企業秘密を教材に載せない。 |

## 画面ラフ（G4・最低2画面）

### 画面1: 応募企業一覧

- **目的**: 受けたい企業を**追加**し、**どれを編集するか選ぶ**。
- **主要要素**: ヘッダ（アプリ名）、**「企業を追加」**ボタン、**企業名のリスト**（タップで詳細へ）、（任意）更新日の一行。
- **遷移**: ログイン後のホーム → **画面2**（行タップ／新規追加直後も同じ）。

### 画面2: 企業ワークスペース（詳細）

- **目的**: その企業向けに **材料を入れる** → **叩きを生成** → **保存**する。
- **主要要素**: 企業名表示・**戻る**（一覧へ）、**材料**（複数行テキスト）、**「叩きを生成」**ボタン、**生成結果**（編集可能テキスト）、**「保存」**ボタン、（任意）短い注意（デモは架空データで、等）。
- **遷移**: **画面1** → （保存後も）**画面1** へ戻る or 同画面にトースト。

### 画面3（任意・なくても可）

- **設定**（アカウント・ログアウト）だけに絞るなら Week2。Day7 の G4 は **画面1・2** で十分説明できる。

## 発表スライドたたき（コピペ用）

### スライド1: タイトル・誰向け

- 仮題: **就活ES — 材料から叩きを作るメモ帳**
- 誰向け: **28卒・サマーインターン応募が増える自分（同じ悩みの学生にも）**

### スライド2: 課題

- 企業が増えるほど **ESを新規で書く負担**が重い
- 過去のES・メモが **バラバラ**（いまは Google ドキュメント）
- **叩きを早く作りたい**が、毎回メモも含めて作業が増える

### スライド3: 解決の主導線

- **応募企業を追加**（または一覧から選択）
- **材料を貼る**（過去ES・メモ）
- **叩きを生成**（LLM）→ **編集して保存**
- **一覧に戻って**別の企業でも繰り返し

### スライド4: スコープ（選んだ案）

- 選定: **案B（標準）** — 一覧＋詳細の2画面、Supabase で永続化
- **やらない（今回）**: GDoc自動連携、締切・選考管理、設問の完全網羅
- デモデータは **架空の企業名・ダミー文章**

### スライド5: 画面イメージ（ラフの言い換え）

- **画面1**: 企業リストと「追加」
- **画面2**: 材料エリア・生成ボタン・叩き・保存

### スライド6: リスク・メンターに聞きたいこと

- リスク: **生成に時間を取られすぎて保存・一覧が遅れる**／プロンプト次第で品質がぶれる
- 聞きたいこと: **テーブル設計を1枚に抑えるコツ**、**LLMは API Route で十分か**（教材の推奨に合わせたい）
