# UIモックの「型」— 例（4パターン）（**任意の参考**）

**位置づけ**: **Step2** でストーリーと **画面ID・確認できること** が決まっていれば、Step3 は **それに忠実なモック**で足りる。ここでいう **型A〜D** は、レイアウトの **雰囲気・トーンのサンプル**であり、**Step3 で必ず当てはめる必要はない**（迷ったとき・チーム内の共通言語が欲しいとき用）。

**目的（参考として）**: 1枚の静的 HTML で「だいたいこういう密度・構造」の **目安**を揃える。  
**ここでやらないこと**: 本番の Next 実装、API、DB、本物の認証、ピクセルパーフェクトなデザイン。

**機能の個数・難易度・モックに何を付けるか**（ログイン／ダッシュボード／チャット／保存などの整理）は **[`STEP2_FEATURE_SCOPE_AND_MOCK.md`](STEP2_FEATURE_SCOPE_AND_MOCK.md)** を正とする。

**よい粒度の目安**

- **含める**: 画面の「箱」（ヘッダ／メイン／フッタのざっくり）、**主CTA 1つ**、ダミーテキスト、**2画面分**の見え方（切替 or 縦並び）
- **含めない**: ローディングの本実装、バリデーションの網羅、コンポーネント分割の作法、レスポンシブの完璧さ（スマホで「破綻しない」程度でよい）

## 視覚サンプル（静的HTMLモック）

ブラウザで開くと **そのまま操作できる見本**（ダミー文言・切替のみ）。本番実装・API なし。

| 型 | ファイル | トーンの違い（ざっくり） |
|----|----------|---------------------------|
| **A** | [`mocks/mock-type-a-dashboard.html`](mocks/mock-type-a-dashboard.html) | 濃いサイドナビ＋表・業務っぽい密度 |
| **B** | [`mocks/mock-type-b-mobile-column.html`](mocks/mock-type-b-mobile-column.html) | 狭い1カラム・緑系グラデ・大きい主ボタン |
| **C** | [`mocks/mock-type-c-cards.html`](mocks/mock-type-c-cards.html) | オレンジ系・カード＋右下FAB |
| **D** | [`mocks/mock-type-d-wizard.html`](mocks/mock-type-d-wizard.html) | インディゴ系・Step 1→2 の進行バー |

**開き方**: Finder で HTML をダブルクリックするか、Cursor でファイルを右クリック → **Reveal in Finder** → ブラウザで開く。またはローカルサーバー経由でも可。

## 型A — 業務ダッシュボード風（一覧 → 詳細／編集）

**向くテーマ例**: P5 提出物一覧、P3 イベント一覧、社内向けツールっぽい題材。

| 項目 | 内容 |
|------|------|
| **画面1** | 左に簡易ナビ（または上部タブ）、中央に **テーブル風の行リスト**（ダミー5行）、右上に「追加」 |
| **画面2** | 同じヘッダ。メインに **フォーム風のラベル＋入力枠**（複数行）、下に「保存」「一覧へ戻る」 |
| **見た目のトーン** | 情報密度 **やや高め**、余白は詰め気味でもよい |
| **HTML の作り方の例** | `#list` と `#detail` を `display:none/block` で切替、または縦に長く並べてスクロールで見せる |

**これで伝わること**: 「一覧から1件選んで、中で編集する」主導線と、**情報が多い画面でも破綻しないレイアウト**の意思。

## 型B — モバイル単柱＋大きい主ボタン（タスク／記録系）

**向くテーマ例**: P1 メモ・習慣・シンプル記録。

| 項目 | 内容 |
|------|------|
| **画面1** | 幅狭めのカラム（max-width で中央寄せ）、**今日の一行**、その下に **大きな主CTA**（例:「記録する」） |
| **画面2** | テキストエリア＋「保存」、保存後は画面1に戻る導線だけ |
| **見た目のトーン** | **余白多め**、装飾少なめ、1アクションがはっきり |
| **HTML の作り方の例** | 同じファイル内で2セクション。スマホ幅を想定した `max-width: 28rem` など |

**これで伝わること**: 「迷わせない1本道」と、**スマホで触る前提の軽さ**（最終週のレスポンシブ議論のたたき）。

---

## 型C — カード一覧＋投稿（掲示板／軽いSNS風）

**向くテーマ例**: P2 小さな掲示板、コメント付き一覧。

| 項目 | 内容 |
|------|------|
| **画面1** | **カード複数**（タイトル1行＋本文2行のダミー）、フローティング or 下部に「投稿」 |
| **画面2** | タイトル入力＋本文＋「投稿する」、キャンセルで一覧へ |
| **見た目のトーン** | カジュアル、角丸・影は **あってもなくてもよい**（雰囲気が伝われば十分） |
| **HTML の作り方の例** | カードは `div` の繰り返しでよい。画像はなくてよい |

**これで伝わること**: 「一覧で流し読み → 投稿」という **読む／書く** の往復がイメージできる。

---

## 型D — 2ステップウィザード（申込・アンケート・確認フロー）

**向くテーマ例**: P3 申込、P4 アンケート、入力→確認の分かれ目がはっきりする題材。

| 項目 | 内容 |
|------|------|
| **画面1** | 上部に **Step 1 / Step 2** の表示（丸数字でもテキストでも可）、フォーム項目、**次へ** |
| **画面2** | 入力内容の **要約表示**（ダミーでよい）、**確定** と **戻る** |
| **見た目のトーン** | フォーム中心、**進行感**が最重要 |
| **HTML の作り方の例** | 同一ファイルで `#step1` `#step2` を切替。バリデーションは不要 |

**これで伝わること**: 「途中で迷子にならないか」「確認の一手間を入れるか」の **合意**。実装ではステップを1画面にまとめるかは別議論でよい。

## 運用上の使い方（メンター・受講生）

1. **Step2** でストーリーと **画面ID・確認できること** を先に固める（正はここ）。
2. **Step3** の `STEP3_UI_MOCK.html` は **Step2 に忠実**に作る。型 A〜D は **迷ったときの参考**で、**必ずしも1つに当てはめなくてよい**。
3. 雰囲気だけ型に近づけたいとき、上の **視覚サンプル**（`mocks/*.html`）のトーンを借りてよい。
4. チャットで見た目を直すときは **同一 HTML を編集**（Skill: `course-mvp-step3-ui-mock`）。
