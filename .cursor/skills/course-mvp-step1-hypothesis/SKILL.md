# AIコース — Step1（MVP仮説インタビュー）

## 目的

推測でテンプレを埋めず、**質問に答えていく過程**で内容を **`docs/milestone_rubrics.md` の G1〜G3** に載る形へ寄せ、**1ファイルのレビュー用アウトプット**を出す。

**難易度の扱い（この Skill の仕様）**

- **ゴール**: 最終的に **緑帯で現実的な主導線1本**が書ける状態にする（黄は「削定後」か要素を明示して限定）。
- **出し方**: 最後にまとめて削るだけでなく、**インタビュー中**にユーザー発言が `IDEATION_AND_SCOPE_LADDER` の **黄・赤**に触れそうなときは **その場で1問だけ**確認し、**緑に落とせる言い換え**（例: リアルタイムチャット → 掲示板＋手動更新）を提案する。**無理に肯定しない**。
- ユーザーが列挙した機能が**多すぎる**場合は **メンター相当の直言**と**削る優先順**で緑に収める（嘲笑しない・理由はラダーに基づく）。

## 先に読む（この Vault／リポジトリにあるとき）

- `docs/milestone_rubrics.md`（G1〜G3）
- `docs/DAY1_MVP_TEMPLATE.md`（見出しを出力に合わせる）
- `docs/IDEATION_AND_SCOPE_LADDER.md`（①の問い・P1〜P5・緑/黄/赤）

標準スタックの前提は `docs/STACK_CHOICE.md` と `.cursor/rules/curriculum-glossary.mdc` に合わせる。

## 進め方（必須）

1. **最初**: ユーザーがすでに持っているメモがあれば短く聞く。なければ「テーマは何か」だけ確認する。
2. **インタビュー**: 次の順で **1テーマずつ**。曖昧なら掘る。長文はユーザーに任せ、エージェントは質問と要約に徹する。
   - **誰**: 具体的な一人（「みんな」禁止に近い）
   - **状況**: いつ・どこで・何が起きているか
   - **課題**: つらさを**行動**に落とせるか（例: 毎回○分かかる）
   - **既存の代替**: いまは何で凌いでいるか
   - **最小の成功**: 2週間後、一人が喜ぶのは何か（1行でも可）
3. **機能の洗い出し（削りの入力）**: ユーザーに「いま頭にある**やりたいこと・機能**を箇条書きで全部」と必ず聞く（すでに言っていても、漏れがないか確認）。**主導線以外**も含めて列挙させる。
3.5. **インタビュー中の難易度ゲート（先に止める）**: ユーザーの発言・列挙の**たびに**、次に当てはまるかを見る。当てはまるなら **確認質問を1つ**入れ、**緑で足りる代替**を一言提案してから次へ進む（長引かせない）。

| 聞こえたことの例 | 確認する一言（例） | 緑寄りの言い換えの例 |
|------------------|-------------------|----------------------|
| チャット、同時、リアルタイム | 「常時つながってる必要ありますか？ それとも投稿が見れればよい？」 | 掲示板・コメント一覧で十分にしないか |
| 管理者・権限が複数 | 「ロールは何種類必要？ まずは自分だけでよい？」 | 1ユーザー＋DB行の自分だけ、に寄せる |
| 外部と深く連携（決済・在庫・地図本格） | 「なしでダミー表示だけでも検証できる？」 | 手入力・CSV1本・読むだけAPI など |
| 通知を常時 | 「アプリ内の一覧で十分？ プッシュ必須？」 | 画面内バッジ・メールは最終週以降 |
| 画面が増えそう | 「最初の1週間で触ってほしい画面は2つで足りる？」 | 一覧＋詳細に絞る |

4. **スコープ帯**: `IDEATION_AND_SCOPE_LADDER` の **P1〜P5のどれに近いか**（なければ同等の重さを一言）。**緑 / 黄（削定後）** を宣言。黄要素があれば列挙。**3.5 で既に緑寄りに言い換えた内容**をここに反映する。
5. **過多チェック（メンター相当）**: 列挙された機能を見て、次を判断する。
   - **画面がおおよそ3枚を超える**、**主導線が複数**に見える、**黄の典型**（リアルタイム・本格権限・深い外部連携など）が**複数**ある → **「機能が多すぎる／この期間では重い」**と**はっきり書く**（例: 「そのままでは実装単位が○つあり、主導線が分岐しすぎています」）。
   - 緑の **P1〜P5** のどれか**1本の主導線**に落とすなら、**何を残し何を削るか**を優先度つきで提案する。**削ったあと**の一言（G1）と主導線（G2）を**確定案として**書き直す。
6. **G2**: 主導線を **最初の操作 → ゴール** まで箇条書き（分岐は最小）。**削り後案**を正とする。
7. **G3**: **ログイン OR DB** のどちらで満たすか、理由つきで決める（迷ったら教材の標準スタックに寄せる方針を一文添える）。

ユーザーが「まだ全部欲しい」と言う場合は、**緑に収まる最小の1本**と、**削らない場合のリスク（黄・間に合わない）**を1〜2行で対比し、**運営上の推奨は削る方**と明示する。

## やらないこと

- インタビューを飛ばしてテンプレだけ渡さない（時間がないときは「最小5問」に圧縮は可）。
- ユーザーの確認なしにプロダクト仕様をでっち上げない。
- 秘密情報（実データ・本番鍵）を前提にした設計を推奨しない（`docs/MVP_CHECKLIST.md` の禁止と整合）。
- 個人を攻撃する言い方はしない。**機能と期間・帯**に対して直言する。
