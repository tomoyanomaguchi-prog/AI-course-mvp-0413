# AIコース — Step3（外側 UI モック＋改善ループ）

## 目的

`docs/mvp/STEP2_STORY_SCENARIO.md`（**Step2 完了**）を入力に、**静的 HTML 1枚**（`docs/mvp/STEP3_UI_MOCK.html`）で画面モックを作る。**初回作成で終わりにしない**。**改善点のインタビュー → チャット指示でモック更新 → 再インタビュー**を繰り返し、**ユーザーが了承（「いいね」「これでOK」等）するまで**続ける。

本番の Next・API・DB は書かない。**G4**（画面が説明できる）向け。

## 全体の流れ（必ずこのループ）

```
[1] 初回モック作成（STEP3_UI_MOCK.html）
      ↓
[2] 改善点インタビュー（下の「改善インタビュー質問」を使う）
      ↓
[3] ユーザーが「ここをこうして」と指示 → 同一 HTML を編集
      ↓
[4] 再度 [2]（改善点インタビュー）
      ↓
[5] ユーザーが了承するまで [2]〜[4] を繰り返す
      ↓
[6] 終了宣言（Step3 完了・このモックで Day7 の G4 説明に使える状態）
```

**終了の合図（例）**: 「これでいい」「OK」「いいね」「もうない」「了承」「このモックで進める」など。曖昧なら **「他に直したいところはありますか？なければ Step3 完了とします」** と1問確認してから締める。

## 先に読む

- **必須入力**: `docs/mvp/STEP2_STORY_SCENARIO.md`（§4 **画面ID** と **確認できること**・§8 Step3 引き渡し）
- `docs/mvp/STEPS_MVP_FLOW.md`
- モックの範囲: `docs/mvp/STEP2_FEATURE_SCOPE_AND_MOCK.md`
- `docs/milestone_rubrics.md`（G4）
- **（任意）** 雰囲気の参考: `docs/mvp/STEP2_MOCK_TYPE_EXAMPLES.md`・`docs/mvp/mocks/*.html` — **型 A〜D はラベル付きのサンプル**であり、Step3 で **必ずしも当てはめる必要はない**（Step2 でストーリーと画面が決まっているなら、**それに忠実なレイアウト**を優先する）。

## レイアウトの正（型は任意）

- **正**: Step2 の **画面ごとの「確認できること」** と **主導線**。HTML は **S1 / S2 … の情報が読み取れる**ことが最優先。
- **任意**: 型 A〜D（`docs/mvp/STEP2_MOCK_TYPE_EXAMPLES.md`）は **トーンやコンポーネントの置き方の参考**。近いものがあれば借りてよいが、**無理に型に収めない**。ループ中に見た目だけ変える場合も、**ストーリーと矛盾しなければ**よい。

## 進め方（フェーズ別）

### フェーズ A — 初回モック

1. Step2 の **画面ID（S1/S2…）** と **確認できること** を読み、HTML に対応づける。
2. **`docs/mvp/STEP3_UI_MOCK.html`** を作成（ユーザーが保存依頼したときのみ）。**1ファイル完結**。`app/` に触れない。
3. 初回ができたら **すぐフェーズ B に入る**。「できました」だけで終わらない。

### フェーズ B — 改善点インタビュー（毎ループ）

ユーザーに **短く**（長文にさせない）次を聞く。不要な項目はスキップ可。

| # | 聞くこと（例） |
|---|----------------|
| B1 | このモックを見て、**主導線**（最初の操作→ゴール）は Step2 のストーリーと **同じ流れに見える**か？ ズレがあればどこか。 |
| B2 | **各画面で確認したい情報**は足りているか？ 足りないブロックは？ |
| B3 | **トーン**（業務っぽい／カジュアル／余白）は意図に合うか？ |
| B4 | Day7 でメンターに **口頭で説明しづらい** ところはあるか？ |
| B5 | **色・文字サイズ・ボタンの位置**など、すぐ直せそうな見た目の希望は？（Step2 のスコープを膨らませない範囲で） |

回答を受けたら、**反映できるものは `STEP3_UI_MOCK.html` に反映**（ユーザーが保存を依頼したとき）。反映できないもの（本番実装が必要なもの）は **Step2 のストーリー変更が要る**と区別して言う。

### フェーズ C — チャット指示での更新

- ユーザーが「ヘッダーを薄く」「ボタンを右下に」などと言ったら **同一ファイル** を編集。
- 更新後、**必ずフェーズ B に戻る**（短くてよいので「直したので、B1〜の観点でまだ気になる点は？」）。

### フェーズ D — 終了

- 了承の合図が出たら、**1行でサマリー**（例: 「Step3 完了。`STEP3_UI_MOCK.html` v3 を最終とする」）。
- 任意: ユーザーが求めたら、変更履歴を `docs/mvp/STEP3_UI_MOCK_CHANGELOG.md` に追記する案を出す（必須にしない）。

## やらないこと

- Step2 なしでモックだけ作らない。
- **1回の初回作成だけで会話を打ち切らない**（了承までループを前提にする）。
- Next.js 本番コード・API・`.env` 実値。
- Bolt を本番リポの正としない。
- ユーザーの了承なしに無限に質問し続けない（**終了確認の1問**は必ず入れる）。
